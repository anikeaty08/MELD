## Fix 1 — Geometry Loss: Re-encode Anchors Through Current Model

### What is currently happening
`_geometry_loss()` in `updater.py` at line 421 does this:

```python
anchors = torch.from_numpy(anch_np).to(device=device, dtype=dtype)
current_logits = model.classifier(anchors)
```

It loads the stored anchor embeddings as raw numpy arrays and feeds them
directly into `model.classifier()` — skipping `model.backbone` entirely.
This means the geometry loss is comparing classifier outputs for embeddings
that were computed by the OLD backbone (at snapshot time) against what the
CURRENT backbone would produce.

When the backbone updates, the embedding space rotates and stretches. The
stored anchor vectors no longer correspond to meaningful points in the current
embedding space. The KL divergence between old-snapshot logits and
current-classifier logits on stale embeddings is measuring garbage — it is
penalising changes in the classifier that are actually correct adaptations
to the new embedding space.

### Why it matters for the PS
The geometry loss is the mechanism that prevents catastrophic forgetting of
old-class structure. If it is computing on stale embeddings, it provides no
useful gradient signal. This directly causes the `geometry_loss_per_epoch`
values of 1e-5 seen in the runs — the loss is near-zero not because geometry
is well-preserved but because the computation is wrong.

### The fix
The anchor embeddings should be re-encoded through the current backbone at
geometry loss computation time. This is more expensive but produces a
meaningful signal.

File: `meld/core/updater.py`, method `_geometry_loss`, around line 415.

Change:
```python
# WRONG — uses stale snapshot embeddings
anchors = torch.from_numpy(anch_np).to(device=device, dtype=dtype)
current_logits = model.classifier(anchors)
```

To:
```python
# CORRECT — re-encode through current backbone
anchor_inputs_np = snapshot.class_anchor_inputs.get(cid)
if anchor_inputs_np is not None:
    # Use stored raw inputs re-encoded through current backbone
    anchor_inputs = torch.from_numpy(anchor_inputs_np).to(device=device, dtype=dtype)
    with torch.no_grad():
        fresh_embeddings = model.embed(anchor_inputs)
    current_logits = model.classifier(fresh_embeddings)
else:
    # Fallback: use stored embeddings (old behaviour, less accurate)
    anchors = torch.from_numpy(anch_np).to(device=device, dtype=dtype)
    current_logits = model.classifier(anchors)
```

This requires adding `class_anchor_inputs: dict[int, np.ndarray]` to
`TaskSnapshot` in `interfaces/base.py` and storing the raw input tensors
(not just embeddings) in `snapshot.py` during capture. Store at most 20
samples per class — same as current anchor count.

### Alternative if storing raw inputs is too expensive
Use the stored class Gaussian statistics to sample fresh embeddings from the
current model's distribution, then compute the geometry loss against those.
This is already partially done (the `gaussian_samples` path in the current
geometry loss), but the teacher logits are computed using stored classifier
weights instead of re-encoding. The teacher logits path is correct but the
current student path is using stale embeddings.

## Fix 2 — NaN Guard on Fisher Diagonal

### What is currently happening
`snapshot.py` at lines 159-174 already has NaN guards:
```python
fisher_diagonal = np.clip(
    np.nan_to_num(fisher_diagonal, nan=0.0, posinf=1e6, neginf=0.0),
    ...
)
```

This is present and correct. The NaN cascade seen in earlier runs was caused
by `freeze_bn_stats=True` with an unconverged model, which has been identified
separately. The Fisher NaN guard itself is implemented.

### What is still missing
The guard clips to `1e6` for infinite values. A Fisher eigenvalue of 1e6 will
make `pre_bound = 1e6 * lr * sqrt(steps * dim)` which is astronomically large,
and will trigger `BOUND_EXCEEDED` immediately on every task, skipping all
delta updates.

The correct behaviour is: if Fisher is anomalously large (unconverged model),
the oracle should say "I cannot reliably estimate the bound" rather than
producing a huge number that blocks training.

File: `meld/core/snapshot.py`, around line 181.

Change:
```python
diag_eig_max = float(np.max(fisher_diagonal)) if fisher_diagonal.size else 0.0
fisher_eigenvalue_max = max(diag_eig_max, float(kfac_eig_max))
```

To:
```python
diag_eig_max = float(np.max(fisher_diagonal)) if fisher_diagonal.size else 0.0
raw_max = max(diag_eig_max, float(kfac_eig_max))
# Soft-cap: values above 1000 indicate unconverged model or numerical explosion.
# Use 1000 as a conservative but finite upper bound to prevent oracle blackout.
fisher_eigenvalue_max = float(np.clip(raw_max, 0.0, 1000.0))
```

Also add to `oracle.py` in `pre_risk_estimate`:
```python
# If Fisher is at the soft cap, flag the estimate as unreliable
is_fisher_saturated = snapshot.fisher_eigenvalue_max >= 999.0
estimate = OracleEstimate(
    value=value,
    bound_type="empirical_spectral",
    calibrated=False,
    bound_is_formal=False,
    fisher_saturated=is_fisher_saturated,  # add this field to OracleEstimate
)
```

## Fix 3 — Adaptive Geometry Refresh Interval

### What is currently happening
`GeometryConstrainedUpdater.__init__` sets `self.geometry_refresh_interval = 50`
and never changes it. The geometry loss is computed every 50 batches.

With a small dataset (task data = 10,000 / 5 tasks = 2,000 samples, batch_size=128,
steps_per_epoch = 16), 50 batches means the geometry loss fires once per 3
epochs. With 2 epochs total it fires at most once — which is why the geometry
loss is so small it barely matters.

### Why it matters
If refresh_interval > steps_per_epoch, the geometry loss fires fewer than once
per epoch. It effectively does not constrain the backbone at all.

### The fix
Set `geometry_refresh_interval` adaptively: it should fire at minimum once per
epoch, regardless of epoch length.

File: `meld/core/updater.py`, in the `update()` method setup, around line 120.

Add after computing `steps_per_epoch`:
```python
steps_per_epoch = max(1, len(new_data_loader))
# Geometry refresh: fire at least once per epoch, at most every 50 steps.
# For small tasks (few batches), this ensures geometry loss actually runs.
effective_refresh = min(self.geometry_refresh_interval, max(1, steps_per_epoch))
```

Then replace all `bidx % self.geometry_refresh_interval` with
`bidx % effective_refresh`.

## Fix 4 — Flip pretrained_backbone Default to True

### What is currently happening
`api.py` line 21: `pretrained_backbone: bool = False`

When `False`, ResNet32 starts from random initialisation. On CIFAR-10 this
requires 50+ epochs to reach good embeddings. The geometry loss and Fisher
information are both meaningless until embeddings are stable.

With `pretrained_backbone=True`, `backbone.py` maps ImageNet ResNet-18 weights
into the CIFAR ResNet (the `_init_from_torchvision` method already exists).
This gives much better initial embeddings, so:
- Fisher is meaningful from epoch 1
- Geometry loss computes real signal from the start
- The oracle pre-bound is calibrated correctly
- Fewer epochs are needed (10-15 instead of 50)

### The fix
File: `meld/api.py`

Change:
```python
pretrained_backbone: bool = False
```
To:
```python
pretrained_backbone: bool = True
```

Note: the `_init_from_torchvision` method in backbone.py already handles
this correctly. It maps ResNet-18 ImageNet weights into the CIFAR ResNet
architecture. This is not a trivial change in the codebase — it's a single
default flip that activates already-written code.

The only reason to keep `False` is if you want to demonstrate convergence
from scratch, which is relevant for the compute savings claim. The solution:
keep `False` as default for the `geometry` strategy but flip to `True` for
the `frozen_analytic` strategy where the backbone must be strong from the start.

## Fix 5 — PAC Bound as Formal Pre-training Gate

### What is currently happening
The pre-training gate in `runner.py` line 209 uses the empirical spectral
estimate:
```python
if task_id > 0 and pre_risk_estimate.value > self.config.bound_tolerance:
```

`pre_risk_estimate` is `empirical_spectral` — a heuristic. If this exceeds
`bound_tolerance=10.0`, the update is skipped.

The `pac_style_gap` and `pac_equivalence_bound` ARE computed (lines 901-913)
but they are only stored in the results JSON. They are never used as a gate.

### Why it matters for the PS
The PS asks for "mechanisms to detect distribution shifts" and "quantify
equivalence with theoretical bounds." Using a heuristic as the gate while
the formal bound sits unused in the JSON is backwards — the formal bound
is the one that should control whether to deploy.

### The fix
Add a `pac_gate_tolerance` parameter to `MELDConfig` and use the PAC bound
as a secondary gate alongside the spectral estimate.

File: `meld/api.py`:
```python
pac_gate_tolerance: float = 0.1  # skip if pac_style_gap > this (10% loss gap)
```

File: `meld/benchmarks/runner.py`, after computing pac bounds (around line 901):
```python
# PAC formal gate: skip if the theoretical gap exceeds tolerance
pac_gate_triggered = False
if (pac_style is not None
    and pac_style.bound_is_formal
    and pac_style.value > float(getattr(self.config, "pac_gate_tolerance", 0.1))):
    pac_gate_triggered = True
    # Log that PAC gate triggered, recommend collecting more data
```

Then use `pac_gate_triggered` alongside `pre_risk_estimate.value` to determine
whether to proceed with the delta update or recommend full retrain.

This makes the PS claim "formal guarantee-gated deployment" actually true in
the code, not just in the JSON output.

## Fix 10 — Layer-Wise Fisher (Populate K-FAC Fields Properly)

### What is currently happening
K-FAC factors ARE being computed in `snapshot.py` for the last 2 linear layers.
`kfac_factors_A` and `kfac_factors_G` are populated. The EWC loss in
`updater.py` does use K-FAC via:
```python
pen = pen + (G @ dW @ A * dW).sum()
```

So layer-wise Fisher IS implemented and used. The PS claim "layer-wise Fisher
information" is already satisfied.

### What is missing
The K-FAC factors are only computed for 2 layers. A complete implementation
would cover all layers with significant parameters. Currently 90%+ of the
model's parameters use diagonal Fisher, not K-FAC.

### The improvement
In `snapshot.py`, change the K-FAC selection from "last 2 linear layers" to
"all layers above a parameter count threshold":

```python
# Current: only last 2 linear layers
linear_modules = [m for m in model.modules() if isinstance(m, nn.Linear)]
kfac_modules = linear_modules[-2:] if len(linear_modules) >= 2 else linear_modules

# Better: all linear layers with > 1000 parameters
kfac_modules = [
    m for m in model.modules()
    if isinstance(m, nn.Linear) and m.weight.numel() > 1000
]
```

This increases K-FAC coverage from 2 layers to typically 4-6 layers in ResNet32.

## Fix 11 — Gradient Projection (PCGrad) — Already Implemented

### Status: ALREADY DONE
PCGrad gradient projection is implemented at `updater.py` lines 284-307.
`projected_step_fraction` is tracked and reported.

### The real problem
As diagnosed: `projected_step_fraction=1.0` means PCGrad fires every step,
meaning CE and regularisation gradients conflict every step.

The fix is not to change the PCGrad implementation but to ensure the base
model is converged before regularisation is needed. With a converged model,
CE gradients are small (near loss minimum), conflicts are rare, and
`projected_step_fraction` should be 0.05-0.15.

### One actual missing piece
PCGrad currently projects regularisation away from CE when there is a conflict.
This is the right default for PS requirement 1 (preserve task performance).
But it means if regularisation is consistently losing conflicts, it never
updates. Add a conflict frequency monitor:

```python
# If projected_step_fraction > 0.8, emit a warning
if projected_steps / max(1, total_steps) > 0.8:
    warnings.warn(
        "PCGrad projected >80% of steps. "
        "Regularisation has minimal effect. "
        "Consider increasing lambda_geometry/lambda_ewc or training base model longer.",
        stacklevel=2,
    )
```

This turns a silent failure into a visible warning.

## Fix 13 — Formal PAC Bound as Pre-Training Gate (detailed)

This is Fix 5 from Phase 1 expanded. The PAC gate should work as follows:

```
Before training task T:
  1. Compute spectral estimate (existing): risk = λ_max * lr * sqrt(steps * d)
  2. Compute PAC Hoeffding gap (existing): ε = sqrt(log(1/δ) / 2n)
  3. Compute PAC importance-weighted bound (existing): ε_IW = ...

  Gate logic:
    if risk > bound_tolerance:
        → BOUND_EXCEEDED (skip, full retrain)
    elif ε_IW > pac_gate_tolerance (e.g. 0.10):
        → BOUND_EXCEEDED with reason "pac_formal_gate"
    else:
        → proceed with delta update

After training task T:
  4. Compute drift_realized_post
  5. If drift_realized_post > risk (bound violated):
        → BOUND_VIOLATED (still deploy cautiously, trigger retrain soon)
  6. If pac_equivalence_bound < pac_gate_tolerance:
        → SAFE_DELTA with formal_guarantee=True in output
```

The key output addition is `formal_guarantee: bool` in the Decision dataclass —
`True` only when the PAC bound is below tolerance and importance weighting
was actually applied. This is the PS's "formal guarantee" made explicit.

## Fix 14 — Auto-Tune lambda_geometry from Fisher Spectrum

### What is currently happening
`auto_config.py`'s `derive_train_config()` already computes adaptive lambdas:
```python
lambda_geometry = (1.0 + scale) / fisher_max
lambda_ewc = (fisher_max / fisher_mean) * (0.5 + scale)
```

But `auto_derive_hparams: bool = False` in the config defaults. So this code
never runs.

### The fix
Flip the default:
```python
auto_derive_hparams: bool = True  # was False
```

This activates the existing adaptive lambda computation. When Fisher is large
(model is sensitive), `lambda_geometry` is reduced to prevent over-regularisation.
When Fisher is small (model has converged), `lambda_geometry` is increased.

### Improvement to the formula
The current formula gives `lambda_geometry = 1 / fisher_max`. With
`fisher_max=32.68`, this gives `lambda_geometry=0.031` — very small.
A better formula:
```python
# Ensure geometry lambda is always at least base_lambda * 0.1
lambda_geometry = max(base_config.lambda_geometry * 0.1,
                      (1.0 + scale) / fisher_max)
```

## Fix 15 — Adaptive Fisher EMA Decay Based on Drift Score

### What is currently happening
`runner.py` lines 170-175 already implements this:
```python
ratio = float(last_drift_score) / max(1e-6, float(self.config.shift_threshold))
decay = base_decay / (1.0 + ratio)
self.snapshot_strategy._ema_decay = float(np.clip(decay, 0.05, 0.95))
```

When drift is high (ratio >> 1), decay is reduced (snapshot forgets history
faster). This is correct behaviour.

### What is missing
The drift score used here is the post-training KL drift, but this fires
AFTER the update. The EMA decay for the NEXT task's snapshot is set based on
the PREVIOUS task's drift. This is correct directionally but one task delayed.

A better version would use the pre-training MMD score (which exists in the
results as `pretrain_mmd_score`) to set the EMA decay BEFORE the current
task's snapshot, not after:

```python
# Before snapshot capture for task T:
if pretrain_shift is not None and pretrain_shift.shift_detected:
    # High pre-training shift → reduce EMA decay aggressively
    mmd_ratio = pretrain_shift.score / max(1e-6, self.config.shift_threshold)
    decay = base_decay / (1.0 + 2.0 * mmd_ratio)  # 2x sensitivity
    self.snapshot_strategy._ema_decay = float(np.clip(decay, 0.05, 0.95))
```
